# 贡献指南 (CONTRIBUTING.md)

感谢您对 XPathEditor 的贡献！为保证代码质量与团队协作效率，请在提交 PR 或 Issue 前遵循下列指南。

## 目录

- 分支策略
- 提交信息规范
- Pull Request 流程
- 代码风格与格式化
- 测试
- 持续集成 (CI)
- Issue 报告
- 许可证与版权

---

## 分支策略

- 使用 `master` 作为稳定发布分支，所有发布来自 `master` 的 tag。
- 新功能使用 `feature/` 前缀：`feature/描述`。
- 修复使用 `fix/` 前缀：`fix/问题描述`。
- 紧急修复使用 `hotfix/` 前缀。
- 开发集成分支可使用 `develop`（如项目需要）。

## 提交信息规范

- 请遵循简洁、可读的提交信息。建议使用类似 Conventional Commits 的结构：

  - `feat: 新功能描述`
  - `fix: 修复描述`
  - `docs: 文档更新`
  - `chore: 构建/脚手架/依赖更新`

- 提交正文（如果有）应说明为什么要做出该改动及其影响。

## Pull Request 流程

1. 从 `master` 或 `develop` 创建新的分支。
2. 保持 PR 小而专注，一个 PR 解决一个主题问题。
3. 在 PR 描述中列出改动点、如何测试以及影响范围。
4. 至少一位审阅者批准后方可合并。
5. 合并时优先使用 "Squash and merge" 或按团队约定的合并策略。

## 代码风格与格式化

- 项目包含 `.editorconfig`，请在提交前使用编辑器或格式化工具自动应用规则。
- 命名、缩进与样式应遵循现有代码库约定（代码审查时会检查）。
- 添加或更新代码时，请同时更新或添加相应的单元测试。

## 测试

- 所有关键逻辑应由单元测试覆盖。
- 使用 `pytest`（如项目使用 Python）或项目指定的测试框架运行测试。
- 在提交 PR 前请确保本地通过测试并运行 CI。

## 持续集成 (CI)

- 推送或 PR 时应触发 CI（如 GitHub Actions）。
- CI 应至少包含：安装依赖、静态检查、运行单元测试。

## Issue 报告

- 提交 Issue 时请提供复现步骤、期望行为、实际行为、日志或异常信息及环境信息（OS、Python/运行时版本等）。

## 许可证与版权

- 本项目采用 MIT 许可证。提交代码即表示您同意将贡献纳入该许可证下。
